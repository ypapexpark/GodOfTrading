# GodOfTrading

Bybit Futures(+Binance 듀얼벤뉴) 자동매매 시스템. LaunchAgent(`com.cryptosignal*`)가 로컬 워킹트리의
Python 파일을 5분/3분 주기로 직접 실행한다 — 별도 빌드/배포 단계 없이 **파일을 고치면 다음 스캔부터
바로 라이브에 반영**된다. 주요 파일: `main.py`(스캔/게이트/진입 로직), `trader.py`(포지션 관리/SL/청산),
`config.py`(리스크 파라미터), `trade_state.json`(실시간 상태 — 봇이 계속 씀).

## 작업 전 — 토큰 효율화

- 로그(`/tmp/godoftrading.log`, `/tmp/godoftrading_err.log`)나 큰 diff를 통째로 읽지 말고
  `grep`/`tail -n`/`sed -n`으로 필요한 구간만 잘라서 본다.
- 여러 파일을 뒤지는 탐색성 조사(예: "이 로직 어디서 쓰이나")는 Explore 서브에이전트에 위임해서
  원본 출력이 아니라 요약만 메인 대화로 가져온다.
- 이미 이번 세션에서 읽은 **정적 코드 파일**은 다시 읽지 않는다. 단, 아래 예외를 반드시 지킨다.
- 대화가 길어져서(로그/diff 반복 인용 등) 무거워지면 새 세션으로 넘기자고 먼저 제안한다.
  이 프로젝트의 핵심 판단 근거는 `~/.claude/projects/.../memory/project_godoftrading.md`에
  저장되어 있으니 새 세션에서도 맥락을 다시 불러올 수 있다.

### 예외 — 효율화 규칙을 적용하면 안 되는 것

봇이 실시간으로 계속 쓰는 파일(`trade_state.json`, `trade_candidates.jsonl`,
`trade_execution_journal.jsonl`, 로그 파일)과 잔고/포지션/일손실 같은 라이브 수치는
**절대 캐시하거나 "아까 읽었으니 됐다"고 넘기지 말 것** — 매 판단 시점에 다시 읽는다.
효율화는 "코드를 어떻게 읽을지"에만 적용되고, "매매 판단에 쓰는 데이터가 최신인지"에는 적용되지 않는다.

## 커밋 전 — 품질 체크리스트

1. 변경한 `.py` 파일 `python3 -m py_compile`로 문법 검증
2. 리스크/게이트 로직을 건드렸다면 관련 함수를 직접 import해서 대표 케이스(정상/경계값)로
   동작을 눈으로 확인 — 이 프로젝트는 실거래 봇이라 "컴파일 됨" != "의도대로 동작함"
3. 규모가 있는 변경이면 `/code-review` 또는 `/simplify`로 한 번 더 검토
4. config.py 파라미터를 바꿀 때는 반드시 **왜 바꿨는지, 어떤 실측 데이터 근거인지**를 주석과
   커밋 메시지에 남긴다 (이 저장소의 기존 관례) — 나중에 왜 이 값인지 추적 가능하게
